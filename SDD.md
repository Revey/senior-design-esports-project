# Spec-Driven Development (SDD)

**Summary**: A documentation-first software engineering methodology for AI-assisted coding. Requires writing explicit specs before any code, making AI agents precise implementers rather than unpredictable generators.

**Sources**: `raw/Spec-Driven Development (SDD) A Comprehensive Guide.md`, `raw/Diving Into Spec-Driven Development With GitHub Spec Kit.md`

**Last updated**: 2026-04-28

---

## The Problem SDD Solves

Unstructured AI-assisted coding ("vibe coding") produces three recurring failures:

| Problem | Description |
|---|---|
| **Context Drift** | The AI forgets earlier rules as the session grows |
| **Architectural Hallucination** | The AI invents libraries or patterns you didn't approve |
| **Technical Debt** | Messy code born from reactive, unstructured prompting |

SDD also prevents the shared-context failure that occurs across teams: without explicit specs, a PM, backend engineer, frontend developer, and designer each make reasonable but incompatible assumptions — a failure that only surfaces when the code is already written.

## The SDD Loop

Every coding session follows this strict three-step cycle. No exceptions.

```
Plan → Implement → Verify → (if changes needed: update spec first) → Plan → ...
```

1. **Plan** — Developer and AI collaborate to write or update a spec. No code is written in this phase.
2. **Implement** — AI reads the spec and writes *only* what is in the spec.
3. **Verify** — Developer tests against the spec's acceptance criteria. If something fails, **update the spec first**, then restart the loop.

**Critical rule:** Never say "fix it" when verification fails. Evaluate *why* it failed. If the spec was ambiguous, clarify the spec, then tell the AI to re-read and fix the implementation.

## The SDD File Ecosystem

Three Markdown files govern the entire process. They serve as both human documentation and direct AI context.

### 1. `CONSTITUTION.md` — The Global Blueprint

Lives at the repo root. Loaded first in every new AI session to anchor context. Contains non-negotiables that the AI cannot override.

**Required sections:**
- **Mission Statement** — one sentence describing what the project does
- **Strict Tech Stack** — explicit list of permitted technologies; actively forbids unapproved frameworks
- **Architectural Conventions** — folder structure rules, ORM-only vs raw SQL, etc.
- **Coding Standards** — type safety rules, error handling patterns, API design conventions
- **Operational Constraints** — security, sensitive data handling, licensing requirements

### 2. `FEATURE_SPEC.md` — The Local Blueprint

Created per feature or module. Lives alongside the code it describes (e.g., `src/features/leaderboard/SPEC.md`).

**Required sections:**
- **Feature Goal** — what this feature achieves
- **User Story** — "As a [user type], I want to [action] so that [value]"
- **Technical Requirements** — frontend (UI layout, state, routing) and backend (schema, endpoints)
- **API Contract** — exact request/response payload structure (JSON examples or Pydantic/Zod schemas)
- **Acceptance Criteria** — boolean checklist defining "done" (e.g., "Table sorting updates without full page reload")

### 3. `IMPLEMENTATION_PLAN.md` — The Execution Strategy

Optional but recommended for complex, multi-file features. Generated by the AI *before* it touches any code.

**Why it matters:** Forces the AI to list every file it will create, modify, or delete. You review this plan and correct any wrong paths or approaches before code is written — catching mistakes at the cost of keystrokes instead of refactoring hours.

## Daily Workflow

```
1. "Read CONSTITUTION.md. We are starting work on a new feature."
2. Write FEATURE_SPEC.md manually.
3. "Read [SPEC.md]. Write an implementation plan detailing the files you will touch."
4. Review the plan. If correct: "Execute the plan exactly as written."
5. Run the code. If it fails: update the spec, then "I updated the spec. Read it and fix the implementation."
```

## GitHub Spec Kit

[[github-spec-kit|GitHub Spec Kit]] is an open-source tool that scaffolds SDD in one command. See that page for setup details. Key additions it introduces:

- **Specify CLI** — `uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>`
- **Slash commands** for supported agents: `/specify` (what/why), `/plan` (how), `/tasks` (breakdown)
- **`constitution.md`** — same concept as `CONSTITUTION.md` above, used to establish opinionated team stacks

## Why SDD Persists Across Sessions

Because `CONSTITUTION.md` and `FEATURE_SPEC.md` are physical files in the repo, a fresh AI session can be fully re-anchored by pointing to them. No session history needed — the specs carry the full project context.

## Benefits Summary

- **Zero context loss** — specs persist across sessions and team members
- **Deterministic outputs** — strict API contracts and tech stacks reduce hallucinatory code
- **Built-in documentation** — architectural docs are a byproduct of the workflow; project is documented by the time it ships
